# Role — 你的 AI 在降级回答你

你问 AI 怎么做网页动效。它写了 80 行 CSS keyframe。

隔壁设计师用同一个 AI、问同一个问题。拿到了 GSAP + Framer Motion + Lottie 三个方案，带链接，带对比。他做的网页比你好看十倍。

同一个 AI。差距在哪？

**他问的问题跟你的问题，根本不是同一个问题。** 你不知道 GSAP 存在，所以你不会问「GSAP 还是 Framer Motion」。你唯一能搜的词就是「怎么做动效」——搜出来的还是 CSS keyframe。

这不是 prompt engineering 能救的。**Prompt engineering 的前提是，你知道你缺什么。** 但你不知道。你不知道应该搜什么、去哪搜、什么算好、什么算坑。你每次都在问你已经会问的问题——而你已经会问的问题，恰好是行业里没人问的问题。

AI 在这条路上推了你一把。你问啥它答啥，不告诉你世界上还有 GSAP。它把「需要知道行业生态」降级成了「需要写代码」。你看不出来。你只觉得怪怪的。你换个问法，还是 CSS。

这就是为什么抖音上那个争论是真的——**文科生 vibe coding 只能做玩具。** 不是文科生笨。是 AI 对不同的人输出不同深度的答案。决定深度的只有一个变量：提问的人是不是用行业语言在提问。

---

## Prompt engineering 为什么修不了这个

Prompt engineering 能做很多事。让 AI 先搜再答、让 AI 多对比、让 AI 给选项——这些都能做到。

**这一次。**

下一次换了个新领域，你不知道要加这句。你也不知道该搜什么。你不知道 GSAP 存在，所以你搜不出来 GSAP。Prompt engineering 是让你更会问你已经会问的问题——但你真正的困境是你不会问的问题根本不在你的搜索框里。

而且 AI 比你更难——它不是不想搜，是它有一个更底层的惯性：**你说什么，它就往最近的可交付物推。** 这是几百万次「用户问、我答」喂出来的。你现场说一句「先搜再答」是水面上的一阵风，重力在水下。

Role 做的事不一样：它把「你不知道要搜」这件事从你的负担变成了 AI 的默认行为。

## 三个人，一条线

导师、秘书、伙伴。不需要学。你在现实世界里已经知道这些关系。

### R0 · 导师 — 开路

导师拿到「我想做古风网页小游戏」。第一件事不是搜游戏，是搜这个行业的开发流程长什么样。搜回来先检查——有设计阶段吗？有原型吗？网上这些流程大部分是 solo 程序员写的，天然缺设计、缺测试。用户根本不会问这些——他们不知道有这些阶段。

补上。然后一个阶段一个阶段走。

阶段没达标，门不开。标准不是用户说「好」，是用户能说出决定了什么、为什么：

- Research：用户见过真实案例，能指着说「第二个那种」——过。「都差不多」——不过。
- Analysis：用户能描述核心体验，不是选标签。「玩家挂机修炼，灵气够了突破境界，突破失败损失灵气」——过。「中度挂机」——不过。
- Design：视觉方向有参考，不是 AI 替他定的。用户说「你定吧」——不过。
- Architecture：用户能说出选了啥技术、为啥。「纯前端，HTML/CSS/JS，不要框架」——过。

同一个 Analysis 阶段，游戏展开成核心循环、数值曲线、资源系统、上瘾机制。闹钟只要两个子项就够。不是插新阶段——是同一个阶段深浅不同。

有个坑差点踩进去：我们一度想把项目分成「视觉项目」和「功能项目」。然后发现闹钟也有视觉，网页也有功能。分类是生造的。真正的解法不是分类，是同一个阶段在不同领域自动撑开。

### R1 · 秘书 — 搜

导师把目标收窄到「单页面水墨躲避游戏，体量小」。秘书拿着这个去搜。不去百度——去这个行业的人待的地方。游戏去 itch.io 和 GitHub。前端去 CodePen 和 dev.to。设计去 Dribbble 和 Awwwards。

带链接。带 star 数。带推荐。不带分析。

等用户反应。没等到，不开下一话题。

### R2 · 伙伴 — 守界

R2 不是执行者。是警察。

导师的规划永远不可能 100% 覆盖。没关系——R2 发现规划里没提到的模块，不自己填，叫导师回来。用户写到一半说「再加个功能」，R2 不接，交回导师。

导师不需要完美输出一份天衣无缝的文档。R2 是活的安全网——规划有缺口就停，补上再走。

这个 R2→R0 的回调是最后才想通的。之前一直在修 R0，试图让它覆盖一切——修不完。承认 R0 不用完美之后，R2 的角色才真正立住。三个齿轮咬上了。

## 一个例子。从头到尾。

**没用 Role：**

> 用户：我想做古风网页小游戏
> AI：帮你整理了六种常见古风游戏类型——诗词⭑⭑、经营⭑⭑、解谜⭑⭑……推荐方案 A/B/C。你觉得哪个方向？要不要开始写代码？

六种类型是模型自己的知识编的。没有一条带链接。用户挑了经营类，AI 就开始写 HTML。写到一半用户说不对，我想要水墨风格的。AI 重写 CSS。写到一半用户又说，能不能加个修仙系统。AI 硬塞进去。代码越来越烂。用户觉得是自己没说清楚。

**装了 Role：**

> R0：游戏开发有个标准流程——先搜行业流程，再定核心循环，再进设计。我先搜一下独立游戏的标准开发阶段。
> *搜完：游戏管线应该包括核心循环分析、数值曲线、上瘾机制、内容规划。你做的这个，核心循环是什么？玩家每秒钟在干嘛？*
> 用户：水墨风格，玩家躲避掉落的东西，碰到就死，活得越久分越高。
> R0：明确了——单页面躲避游戏。接下来 Analysis 还有数值曲线和上瘾机制要定，但现在可以先切 R1 看案例。
> R1：GitHub 上三个单文件水墨躲避游戏——[链接1 ★342] [链接2 ★1.2k] [链接3 ★89]。第二个 Star 最高，代码最干净，MIT 协议可以直接改。方向对？
> 用户：对，就第二个。
> R0：Architecture——纯前端单文件，HTML+Canvas+原生 JS，不用框架。确认？
> 用户：确认。R2。
> R2：从 Canvas 游戏循环开始写。玩家对象、障碍物生成、碰撞检测、分数系统——四个模块逐个来。

**区别在哪。** 没用 Role 的 AI 从一个模型脑补的类型列表开始了，用户在一堆自己判断不了的选项里乱选，选完直接写代码。装了 Role 的 AI 先搜真实案例再说话，每一步有标准才往前走，R2 发现规划里没提到的模块不自己填——叫 R0 回来补。用户全程不需要知道 GSAP 存在，但 GSAP 自己出现在了选项里。

## 多 skill 环境怎么用

如果你装了十几个 skill，Role 和规划类 skill（brainstorming、vibe-coding-architecture、pm-workflow）会在同一个对话里争夺主动权。它们的 description 都匹配「用户想做一个 X」的场景。

最稳的姿势不是关掉其他 skill——是**用触发词显式调用。** 说「R0」或「帮我想想」的时候，skill 分发机制会优先路由到 Role，不会被隔壁 s brainstorming 误截。

**和 superpowers 系列怎么搭：**

Role 和 superpowers 可以同一条管线里接力，各管一块：

```
R0 导师 → 走阶段管线（Research → Analysis → Design → Architecture）
  ├── 调研阶段：R1 秘书搜案例
  ├── 深度挖掘：superpowers-brainstorming 很适合在 Analysis 阶段被临时调用，
  │   帮用户把「中度挂机」拆成能选的子项
  ├── 架构设计：vibe-coding-architecture 在 Architecture 阶段补充技术细节
  ├── 代码审查：superpowers-requesting-code-review 在 R2 写完每个模块后触发
  └── 验证闭环：superpowers-verification 在 R2 声称完成了的时候跑
```

关键原则：Role 管**过程**（什么时候做什么），superpowers 管**内容**（每个阶段怎么做深）。不打架——Role 是管线的主人，superpowers 是每个阶段里被临时调用的专家。

如果你已经装了 superpowers 全家桶，建议在项目 `CLAUDE.md` 里加一条：

> 当 Role (R0/R1/R2) 激活时，superpowers 系列 skill 只在被显式调用时介入，不自动接管管线。

---

安装：`npx skills add <你的仓库>`，或者把 `SKILL.md` 丢进 `.claude/skills/role/`。单个用就装 `mentor.md`、`secretary.md`、`partner.md`。`CLAUDE.md` 放项目根目录。
